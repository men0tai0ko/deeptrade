## 1. 戦闘ダメージ式

### 1-1. プレイヤー→敵

```
dmg = max(1, floor(str × 2 × variance - def × 0.7))
variance = 0.85〜1.15
```

**設計意図**: str > def/1.4 であれば正のダメージが出る線形式。strがdefを上回るほどダメージが線形に増加し、最低でも1dmgを保証。

### 1-2. 敵→プレイヤー

```
dmg = max(1, floor(str² × K / (def + str × K)))   K=5
```

**設計意図**: 旧式（減算式）ではdefがstrを超えると常に1ダメになりタンクビルドが無意味になった。除算式にすることでdefが高くてもstrに比例した下限を保証。K=5はstrとdefのバランスポイント。

**具体例（K=5）:**

| 敵str | プレイヤーdef | ダメージ |
|---|---|---|
| 50 | 10 | 42 |
| 50 | 30 | 31 |
| 50 | 50 | 25 |
| 100 | 30 | 87 |
| 100 | 80 | 64 |

### 1-3. クリティカル

```
critRate = min(60, 5 + luk×0.2 + critUpBonus(+5%))
critMul  = 1.8
```

**設計意図**: luk194での上限を約44%に抑制（旧83%から大幅削減）。60%の硬キャップにより「ほぼ毎回クリティカル」という状態をなくし、lukビルドの強さを維持しつつ乱数の意味を保つ。

### 1-4. 防御行動

```
防御時: 被ダメ × 0.5（50%軽減）
```

### 1-5. 強化ボーナス

```
HP加算 = enhLv × 8（ステータスには影響しない）
```

**設計意図**: str/defへの強化ボーナスを廃止し、HPのみに絞った。装備ロスト廃止後の戦略性としてHP強化特化ビルドを差別化。

## 2. 敵スケール

### 2-1. 通常敵

```
scale = (1 + depth×0.05 + floor×0.08) × ngMul × wrMul
ngMul = 1 + ng × 0.5
wrMul = 1 + worldRank × 0.35

str/def/hp = base × scale
exp       = base × scale × (1 + ng×0.2)
dropRate  = min(85, 65 + ng×5)
```

**設計意図**:
- depth補正（+5%/マス）: 深く潜るほど強くなる
- floor補正（+8%/階）: 階層差を明確に
- ng補正（+50%/周）: 2周目以降はかなりきつい
- wr補正（+35%/転生）: 転生ループの挑戦性を維持

### 2-2. ボス基本ステータス

| ボス | str | def | HP | EXP |
|---|---|---|---|---|
| ゴブリン王（1階） | 22 | 8 | 150 | 200 |
| 石の番人（2階） | 38 | 15 | 320 | 400 |
| 影竜（3階） | 58 | 25 | 700 | 750 |
| 沼の蛇神（4階） | 80 | 35 | 1200 | 1200 |

ボスにも同じngMul/wrMulスケールを適用。

**S31調整の背景**: 旧str（50/100/200/350）は被ダメ式 `str²×5/(def+str×5)` の特性上defに無関係に致死ダメージになっていた。str削減によりdefが実効的に機能する範囲に収め、戦闘ターン数を8〜17発生存（回復薬なし）に改善。

| 階層 | luk補正 | 大成功 | 成功 | 失敗 | 大失敗 |

## 8. 販売時間

```
base     = rarityBase × unidMul(未鑑定=2.5)
multi    = typeMulti[type]
lvBonus  = min(0.5, (shopLv-1) × 0.07)
demandMul = 2.0 - demand/100   （demand100→×1.0, demand0→×2.0）
trendMul = isTrending ? 0.65 : 1.0
rand     = 0.8〜1.2（±20%ランダム）
speedMul = max(0.3, 1 - staffSpeedBonus/100)

duration = floor(base × multi × (1-lvBonus) × demandMul × trendMul × rand × speedMul)
```

## 14. セット効果の数値設計

### 基本方針
- 2部位効果：単純なステータス加算（気軽に組める）
- 3部位効果：戦略的な効果（逃走率・クリ倍率等）
- 最大部位効果：強力だがフルセットのコストを要求

| セット | 設計意図 |

| 鉄の覇者6部位 | 装備スロットの半分を鉄で固めるため高コスト。被ダメ-10%は実効的 |

| 幸運の申し子クリ率+8% | luk依存クリ率と累積。luk高い状態で装備するとさらに高クリ率 |

## 17. S43 バランス修正（2026-03-22）

### 毒ダメージ修正
```
旧: max(1, ceil(現在HP × 0.05 × poisonMul))
新: max(1, ceil(最大HP × 0.03 × poisonMul))
```
**理由**: 高HPプレイヤーほど毒ダメが大きくなる逆転現象を解消。最大HP基準にすることで成長に比例した安定値に。

### 4階強制毒の遅延付与
- 旧: 潜入と同時に毒状態
- 新: 5マス進んだ後に発症（`_floor4PoisonAt = 5` カウントダウン）

### rageシステム修正
- rageによるフェーズ3強制移行を廃止
- rage満タン時は演出（フラッシュ+ログ）のみ
- フェーズ移行はHP基準（60%/30%）のみに統一
- rage増加率を大幅低減: ×0.12 → ×0.04（ボス被ダメ時）

**理由**: 高火力プレイヤーほど早くフェーズ3になる逆説的バランス崩壊を解消。

## 18. S44 バランス修正（2026-03-23）

### generateEnemy poisonRate/burnRate バグ修正
- S44以前: `generateEnemy()` の return に `poisonRate`・`burnRate` が含まれておらず、全階層で敵からの毒付与・熱傷付与が完全に無効だった
- 修正後: スプレッド構文で `base.poisonRate` / `base.burnRate` を条件付きコピー

### inferno_drake burnRate 調整
```
旧: burnRate:60
新: burnRate:40
```
**理由**: poisonRate/burnRateバグ修正により熱傷が有効になるため、連続遭遇による熱傷確定を防ぐため緩和。

### rollRarity ボーナス上限追加
```
旧: bonus = depth×0.5 + luk×0.3 + rarityBonus
新: bonus = Math.min(40, depth×0.5 + luk×0.3 + rarityBonus)
```
**理由**: 高luk深部でのLegendary率が約24%に達していたため上限40でキャップ。luk特化の旨みは維持（上限時約19%）。

### 業炎セット熱傷耐性の適用

- `setBurnDmgBonus` > 0 かつ敵が `_burned` 状態: 与ダメに `burnDmgBonus`%（20%）加算
